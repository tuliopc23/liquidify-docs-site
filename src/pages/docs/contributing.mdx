# Contributing to LiqUIdify

Join our mission to bring Apple's design excellence to the web. We welcome contributions from designers, developers, and accessibility advocates who share our vision.

## Code of Conduct

**Be respectful, provide context, and prefer small focused changes.** We foster an inclusive environment where everyone can contribute meaningfully to web design innovation.

## Getting Started

### Development Setup

1. **Fork and clone** the repository:
```bash
git clone https://github.com/yourusername/liquidify.git
cd liquidify
```

2. **Install dependencies**:
```bash
bun install
```

3. **Run development environment**:
```bash
# Build the library
bun run build:lib

# Run tests
bun run test

# Type checking
bun run type-check

# Linting and formatting
biome check .
```

### 2. Implementation Standards

#### Component Guidelines
```tsx
// Follow this pattern for new components
import { forwardRef } from "react"
import { cx } from "styled-system/css"
import { button } from "styled-system/recipes"

export interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
	variant?: "primary" | "secondary" | "ghost"
	size?: "sm" | "md" | "lg"
}

export const Button = forwardRef<HTMLButtonElement, ButtonProps>(
	function Button({ variant = "primary", size = "md", className, ...props }, ref) {
		return (
			<button
				ref={ref}
				className={cx(button({ variant, size }), className)}
				{...props}
			/>
		)
	}
)

Button.displayName = "Button"
```

#### Styling with Panda CSS
```ts
// Create recipes in libs/components/src/recipes/
import { defineRecipe } from "@pandacss/dev"

export const button = defineRecipe({
	className: "button",
	base: {
		// Base styles following Apple HIG
		display: "inline-flex",
		alignItems: "center",
		justifyContent: "center",
		// ...
	},
	variants: {
		variant: {
			primary: { /* Apple blue */ },
			secondary: { /* Subtle styling */ }
		}
	}
})
```

#### Testing Requirements
```tsx
// Component tests with @testing-library/react
import { render, screen } from "@testing-library/react"
import { Button } from "./button"

describe("Button", () => {
	it("renders with correct variant styles", () => {
		render(<Button variant="primary">Click me</Button>)
		expect(screen.getByRole("button")).toHaveClass("button--variant_primary")
	})

	it("supports custom className", () => {
		render(<Button className="custom-class">Click me</Button>)
		expect(screen.getByRole("button")).toHaveClass("custom-class")
	})
})
```

### 4. Pull Request Process

#### PR Guidelines
- **Keep PRs focused** - one feature or fix per PR
- **Write clear titles** describing the change
- **Include context** in the description
- **Add screenshots** for visual changes
- **Update documentation** for API changes

#### PR Template
```markdown
## Summary
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
- [ ] Unit tests added/updated
- [ ] Manual testing completed
- [ ] Accessibility testing done

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Documentation updated
- [ ] Tests pass locally
```

## Documentation Contributions

### Writing Component Docs
1. **Create MDX files** in `docs-public/components/`
2. **Follow the template** structure:

```mdx
# Component Name

Brief description following Apple HIG principles.

## Usage

```tsx
import { Component } from "liquidify-react/component"

export default function Example() {
	return <Component>Content</Component>
}
```

## Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| variant | string | "primary" | Visual style variant |

## Accessibility

- Follows WCAG 2.2 guidelines
- Supports screen readers
- Keyboard navigation included
```

### Documentation Standards
- **Use subpath imports** in examples
- **Include accessibility notes** for every component
- **Show real-world examples** not just basic usage
- **Cross-reference related components**

## Release Process

### Version Management
We follow semantic versioning:
- **Patch** (1.0.1): Bug fixes, documentation
- **Minor** (1.1.0): New features, backwards compatible
- **Major** (2.0.0): Breaking changes

### Changeset Process
```bash
# Add changeset for your changes
npx changeset add

# Describe your changes and select version bump type
# Commit the changeset with your PR
```
